2A.2-front — Estado "reprocesando…" persistente por orden

Verificado en js/features/control-bl.js, función cblReprocesar(orderNumber, btn) (línea ~640): hoy el único feedback es ssbToast(r.detail, ...) (desaparece solo) + el botón .disabled=true por 15 s a ciegas (setTimeout) — no hay chip persistente, no hay señal en la fila de la lista, y los 15 s no tienen relación con cuándo termina realmente el workflow n8n (que corre en background después del disparo). Si el usuario cambia de orden o recarga, pierde todo rastro de que había un reproceso en curso.

Código real de hoy — js/features/control-bl.js

async function cblReprocesar(orderNumber, btn){ if(btn) btn.disabled = true; try { const data = await cblApiSeguimiento({ action: 'reprocesar_bl', order_number: orderNumber }); const r = data.result || {}; ssbToast(r.detail || 'Reproceso disparado.', r.status === 'disparado' ? 'success' : 'info'); if(btn) setTimeout(() => { btn.disabled = false; }, 15000); // 15s a ciegas, no el fin real } catch(e){ ssbToast('No se pudo disparar el reproceso: ' + (e.message || 'error de red'), 'error'); if(btn) btn.disabled = false; } }

3B.8-interim — Chip PL "s/d" distinto de "falta"

Verificado en js/features/seguimiento.js, función renderDocs(r): la función addSoft() (usada solo para PL) crea el('span','seg-doc off', label) — la misma clase off que BL/FC usan cuando el documento genuinamente falta — y le agrega font-style:italic; opacity:.62 inline. Visualmente el chip PL queda idéntico al de un documento faltante (gris, rayado). Coincide con lo hallado en docs/explore/EXPLORE_B8_G1_2026-07-16.md.

Código real de hoy — js/features/seguimiento.js

const addSoft = (label, title) => { const d = el('span','seg-doc off', label); // ← misma clase que "falta" d.title = title; d.style.fontStyle = 'italic'; d.style.opacity = '.62'; wrap.appendChild(d); }; ... addSoft('PL','Packing List — s/d (se verifica recién al enviar por Mailing)');

✓Notas para la aprobación de John

· B.1-fix — costo del override por fila: el segmentado "Transporte del lote" es la parte barata (1 control nuevo, aplica a todo el pegado). El override por fila (3ª columna M/T opcional) es la parte que decide el alcance real: si preferís sin override (todo el lote es 100% el segmentado, sin excepciones por fila), es más simple de implementar y de explicar — avisá si querés recortarlo antes de que arranque la tanda.
· A.2-front — qué dispara cada transición: este mock muestra los 3 estados visuales, pero no resuelve el mecanismo de "fin real" (el plan lo menciona como "poll al fin real"). Mi lectura: polling liviano (¿cada 5-10 s, timeout ~90 s?) contra bl_controls buscando un created_at más nuevo que el momento del disparo — si nunca llega, un 4º estado "no confirmado — probá de nuevo" en vez de quedar eternamente en "reprocesando…". Es una decisión de implementación, no visual — dejo la pregunta abierta en vez de inventar el número.
· B.8-interim — Opción A vs B: mi recomendación es Opción A (reclasificar a seg-doc fut) por ser gratis y ya coherente con la leyenda existente — pero si preferís que PL se destaque MÁS que CRT/COA (semánticamente son distintos: CRT/COA es "todavía no existe en el sistema", PL es "existe pero se verifica en otro momento"), la Opción B es la que pediste en el enunciado. Ambas conviven con off=falta intacto.
· G.1 y C.1 son informativas, no piden aprobación de diseño nuevo — G.1 ya está en el código (solo falta que lo veas funcionando); C.1 es una tabla de contexto para que el implementador arranque con el texto real en la mano, no una propuesta de copy.
