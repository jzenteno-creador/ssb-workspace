`addSoft()` builds the span as `el('span','seg-doc off', label)` (line 381) — **the same `off` class used for genuinely-missing BL/FC** — plus inline `font-style:italic; opacity:.62`. So visually it reads as the identical gray/dashed "falta" chip, which is exactly the ambiguity the code comment worried about but didn't actually avoid.

**Conclusion for B.8, part 1:** the PL chip is gray for order 118833340 not because of any data state for that specific order — it would be gray for **every** order, always, because `v_operacion_estado` has no PL column and the renderer never queries anything for it.

**2. Mailing's doc detection** — completely different data path:
- `js/features/mailing.js:169`: `DOC_LABELS = { bl_draft:'BL', factura:'FC', packing_list:'PL', ... }`
- `js/features/mailing.js:999-1005` (`runPreview`) calls `apiMailing({ order_number, action:'preview' })` → POST `/api/mailing`.
- `api/mailing.js:1,407-414`: proxies to `process.env.MAILING_WEBHOOK_URL` — the n8n workflow **"Mailing Envío Documentación"** (`kh6TORgRg9R1Shj1`). The preview response's `attachments.found`/`attachments.missing` (consumed in `mailing.js:637-666`) is what drives the doc chips shown in Mailing.
- `grep packing` across `js/` and `api/` returns **only** the `DOC_LABELS` mapping — there is no Supabase column/table anywhere in this codebase named `packing_list`/`doc_pl`. Detection happens **inside the n8n workflow at preview time** (per seguimiento.js's own comment, this is presumably a live Drive lookup), not from any persisted Supabase state.

**Exact divergence point:** Seguimiento's PL signal is a hardcoded UI placeholder (no data source exists yet); Mailing's PL signal comes from a real-time n8n/Drive check done only when a mailing preview is generated for that order. These are architecturally disconnected — Seguimiento cannot "see" what Mailing sees because nothing writes the Mailing/Drive PL result back into `v_operacion_estado` (or any table Seguimiento reads).

**cert-origen.js's consumption is a third pattern** (`cert-origen.js:471-481`, `loadCertOrigen`): it only pre-fills the `#co-orden` **input field** (`oi.value = _po`, line 478) for the "generar certificado" form — it doesn't filter the `loadHist()` recent-20 history list either, but that's consistent with cert-origen's UI (a generate-form + a static recent-history list, not an order-scoped list), so it's not really comparable to the filtering question.

**Fact for G.1:** the `__segPendingOrder` bus and its consumption in `loadMailing` exist and are functionally wired (not silently dropped, not missing) — the gap is specifically that Mailing's handler stops at "select + highlight," never sets `_q`/re-renders a filtered list the way control-bl's handler does. That is the exact, observed point where Seguimiento's and Mailing's deep-link handling diverge.
